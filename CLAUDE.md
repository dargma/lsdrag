# CLAUDE.md — 행동 계약 (진입점)

> 이 파일은 문서가 아니라 **행동 계약**이다. 매 세션 자동으로 읽힌다. 200줄 이내 유지.

## 이 프로젝트가 무엇인가

그래프 없이 GraphRAG급 성능을 내는 사내 RAG. **A-RAG**(agentic RAG)를 엔진으로, 구조 질의용
**Page Index**와 조건부 멀티모달(**VLM**)을 tool로 얹는다. 최종적으로 `/rag` skill로 호출된다.
목적은 **"여기서 검증 → 회사로 이식"**. 그래서 모든 결정은 *"이 경계를 통째로 바꿔치기할 수 있나?"*로 수렴한다.

## 작업 시작 절차 (매번)

1. **`PROGRESS.md`를 먼저 읽어라.** 진행 상태의 유일한 진실의 원천이다.
2. 작업할 모듈의 `docs/NN_*.md`를 읽어라.
3. 선행 모듈의 산출물이 **실제로 존재하는지** 확인하고 시작하라.

## 불변 사실 (변경 금지)

@docs/01_FACTS_arag.md

위 파일의 A-RAG 코드 사실(시그니처·메서드명·반환 타입)은 실제 레포에서 검증된 것이다.
추측으로 바꾸지 마라. 어긋나는 구현은 틀린 것이다. 의심되면 레포를 직접 보고 근거(파일·줄)와 함께 갱신.

## 대원칙 1 — 3단계 엄격 분리 (파싱 → DB 구축 → 검색)

```
[1. 파싱]    문서 → IR                 (Upstage. 사내 파서로 교체 가능)
[2. DB 구축] IR → 인덱스 + Page Index   (임베더·저장소 교체 가능)
[3. 검색]    쿼리 → tool → Reader → 답변  (검색전략·Reader 교체 가능)
```

- 단계 간 연결은 **오직 2개 계약**으로만: **1↔2 = IR 스키마(02)**, **2↔3 = 인덱스/PageIndex 인터페이스**.
- 한 단계가 **다른 단계의 내부를 import하면 거부**하라. 계약만 본다.
- 각 단계는 **단독 실행·테스트 가능**해야 한다(파싱만 돌려 IR 산출물 확인 등). 흐리는 코드는 거부.
- 새 파서/임베더/검색기를 붙일 때 **그 단계 모듈 + config만** 바뀌어야 한다. 안 되면 분리가 깨진 것.

## 대원칙 2 — 두 스택 검증 (모든 모듈 필수, 하나라도 빠지면 미완성)

1. **작업 중 검증 (build-time)** — 짜는 즉시 돌리는 인라인 assert·미니 실행. 틀리면 즉시 고친다.
   이 체크는 버리지 말고 `tests/`에 **회귀 테스트**로 남긴다. 모듈 done 판정 = **Post-hoc 게이트**(완료 정의).
2. **사용 중 검증 (run-time)** — 잘못된 입력·외부 API 실패·빈 인덱스를 **실행 시점에 감지**하는 가드.
   각 단계 경계에서 계약 확인(파서 응답이 IR 계약 만족? 인덱스 비었나? Reader 응답 비었나?).
   실패 시 명확한 에러·폴백. **침묵 실패 금지** — 문제는 항상 사용자가 이해할 말로 드러낸다.
   `doctor.py`가 이 런타임 검증의 **설치판**이고, 쿼리 경로에도 같은 가드를 둔다.

## 대원칙 3 — A-RAG 가지치기 (불필요 요소 삭제)

vendoring 기준은 단 하나: **"우리 파이프라인(파싱→DB→검색) 런타임 경로가 import하는가?"**
- 아니면 가져오지 마라. 이미 가져왔으면 **삭제**하라. (`eval.py`, `batch_runner.py`, 벤치·데모·미사용 tool 등)
- 남길 최소 집합: `core/{context,llm,config}.py`, `tools/{base,registry,semantic_search,keyword_search,read_chunk}.py`, `agent/base.py`.
- **남긴 목록·지운 목록·이유·커밋(`a44de6b`)·라이선스(MIT)**를 `src/agent/vendor/SOURCE.md`에 기록.
- vendor 직접 수정 금지. 필요 시 **바깥에서 상속/확장**. 코어 변경은 `AgentContext` 이미지 추적 추가 1건뿐.

## 대원칙 4 — 직관성 (설치·사용 경험은 요구사항)

- 설치는 `INSTALL.md`를 위→아래로 따라가면 끝. 각 단계 = **한 명령 + 한 검증**. 막히면 "무엇이·왜·다음에 뭘".
- 사용은 **`/rag 질문` 한 줄**. 내부 복잡성(파서·인덱스·tool)을 노출하지 마라.
- 에러는 사람 말로 + "이걸 하면 됩니다". 스택트레이스만 던지지 마라.
- 설정은 한 곳: `config.yaml`(경로·모델) + `.env`(키). 건드릴 값/안 건드릴 값을 주석으로 구분.

## 대원칙 5 — 개발 지시서 ↔ 배포 산출물 분리 (엄격)

이 repo에는 **두 종류**의 파일이 섞여 있다. 절대 뒤섞지 마라.

- **개발 전용 (지금 만들 때만, 배포 금지)**: `CLAUDE.md`, `PROGRESS.md`, `docs/`, `_MASTER.md`, `tests/`, `benchmarks/`.
  → 엔진을 *만드는* 나를 위한 지시서·추적·검증·평가. **배포되는 skill에는 한 줄도 들어가면 안 된다.**
  → `benchmarks/`는 **내부 테스트·평가 전용**(eval 하네스·셋·결과). 엔진/skill 런타임이 import하지 않는다.
- **배포 산출물 (사용자에게 가는 것)**:
  - **skill** = `rag/`(SKILL.md, scripts/) → `~/.claude/skills/rag/`로 복사. **얇은 껍데기.**
  - **엔진** = `src/` + `config.yaml`(+ `.env`) → 사용자가 둔 위치. skill이 config로 가리킨다.
  - 사용자 안내 = `README.md`, `INSTALL.md`.

규칙:
- **`rag/` 안의 어떤 스크립트도 `docs/`·`CLAUDE.md`·`PROGRESS.md`를 런타임에 읽거나 import하지 마라.**
  skill은 `src/`(엔진)와 `config.yaml`에만 의존한다. 개발 지시서 없이도 단독 동작해야 한다.
- skill 패키징 시 개발 전용 파일이 섞이면 그 패키징은 틀린 것이다(09_skill의 패키징 경계 참조).
- "이 지시서가 있어야 동작"하는 코드를 쓰면 분리가 깨진 것이다.

## 경로·시크릿 하드코딩 금지

- skill 자체 위치 → `${CLAUDE_SKILL_DIR}`. 엔진/인덱스/데이터 위치 → `config.yaml`(사용자 지정).
- API 키 → **환경변수**(`UP_TOKEN`=Upstage, `OPENAI_API_KEY`=Reader). 코드·문서·config.yaml 평문 금지.
  `.env`는 반드시 `.gitignore`. 키 노출 시 즉시 폐기·재발급.

## 교체 경계 (이식 시 바꿀 곳, 흐리지 마라)

- `src/parser/adapter.py` — Upstage 응답 → IR. 사내 파서 교체 시 **이 어댑터만**.
- `config.yaml` — 파서/Reader 엔드포인트·모델, `embedding.model`.
- **파서 응답 ↔ IR 경계** — 파서가 무엇이든 출력은 IR로 흡수. 다운스트림은 IR만 본다.
- (마스터의 `services/docling_server/`는 폐기. Upstage가 원격 API라 로컬 파서 서버 불필요.)

## 코드 작성 규칙

- **간결한 명세, 너의 판단.** 문서는 "무엇·왜"만 고정. "어떻게"는 너의 판단. 단 **검증 게이트는 반드시 충족**.
- **검증 게이트가 완료 정의다.** "대충 동작함"으로 넘어가지 마라.
- **모듈 경계 존중.** 한 모듈 작업 중 다른 모듈을 광범위 수정 금지. 인터페이스(IR, tool 계약)로만 연결.
